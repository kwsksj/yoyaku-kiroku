# Cloudflare D1移行における `yoyaku-kiroku` 詳細検討メモ

**作成日**: 2026年3月3日  
**ステータス**: Draft（検討段階）  
**位置づけ**: 実装着手前の詳細検討資料

## 1. 目的

- `CLOUDFLARE_D1_MIGRATION_PLAN.md` の全体方針を、`yoyaku-kiroku` 観点で具体化する。
- 実装前に「何を決めるべきか」「どこが未決か」を明確化する。
- データ移行・切替・ロールバックの判断基準を先に定義する。

## 2. スコープ

### 2.1 対象（In scope）

- 予約・名簿・参加記録の D1 移行方針
- API 契約（読み取り系/書き込み系）
- Spreadsheet から D1 への移行手順
- index JSON 生成の D1 起点化（`participants_index.json` / `schedule_index.json`）
- 切替判定とロールバック手順

### 2.2 非対象（Out of scope）

- `media-platform` 側の詳細仕様（別紙参照）
- `/students-index` / `/tags-index` の配信経路切替判断（別紙で扱う）
- 教室サイト/作家サイトのUIデザイン詳細
- Notion作品運用の詳細

## 3. 現状の課題整理（要確認）

- Spreadsheet/GAS 依存により、読み取り遅延と運用負荷が発生している。
- 予約データの正本が明確でなく、移行時に整合性リスクが高い。
- 切替時の手順（凍結、差分同期、ロールバック）が文書化不足。

## 4. 正本境界（案）

| データ領域 | 正本 | 補助/派生 | 備考 |
| --- | --- | --- | --- |
| 予約・名簿・参加記録 | D1 | Spreadsheet（移行期間は参照） | 書き込みは Worker API 経由に統一 |
| 作品メタデータ・タグ | Notion | `gallery.json` / `tags_index.json` | 既存運用継続 |
| 画像実体 | R2 | なし | 既存運用継続 |
| 参加者/日程 index | D1 由来の二次データ | R2 JSON 配信 | 高速配信用キャッシュ |

## 5. データ移行設計（案）

### 5.1 移行フェーズ（詳細）

1. **棚卸し**
   - Spreadsheet カラムと D1 スキーマの 1:1 マッピング表を確定する。
2. **初回バックフィル**
   - Spreadsheet から D1 に一括投入し、件数/キー/期間を照合する。
3. **移行期間運用**
   - D1 を書き込み正本とし、必要に応じて Spreadsheet へ片方向同期する。
4. **切替**
   - GAS書き込み停止、Spreadsheet参照専用化、監視強化を実施する。

### 5.2 移行時の不整合対策

- 同期ジョブは冪等（同じ入力を複数回実行しても同じ結果）にする。
- 差分検証は日次で実施し、件数差・キー欠落を検知する。
- 予約ステータス変更は監査ログ（イベントテーブル）に追記する。

## 6. API設計（検討項目）

### 6.1 読み取り系（D1直読）

- 空き枠取得
- 予約一覧取得
- 生徒/会員プロフィール参照

### 6.2 書き込み系

- 予約作成
- 予約変更
- 予約キャンセル

### 6.3 API原則

- ブラウザ -> D1 直接アクセスは禁止（Worker API経由）。
- 書き込みAPIは認証必須 + 監査ログ必須。
- リトライ前提のため冪等キーを導入する。

## 7. index JSON 運用（`yoyaku-kiroku` 観点）

### 7.1 方針

- `participants_index.json` / `schedule_index.json` は当面維持する。
- 生成元は GAS から D1 Materializer（Worker/Cron/Queue）へ切替える。
- `/students-index` / `/tags-index` は既存経路を維持し、切替判断は `media-platform` 側検討メモで扱う。

### 7.2 最低要件

- 既存キー互換（`generated_at`, `timezone`, `dates` など）
- 短TTL配信（既存運用相当）
- `generated_at` 監視による鮮度チェック

## 8. 運用・セキュリティ（要件）

- 環境分離: `dev` / `stg` / `prod`
- Secrets は Workers Secrets で管理
- 認証/認可: 会員向けAPIと管理APIの境界を明確化
- 監視:
  - APIエラー率
  - API p95レイテンシ
  - index JSON 鮮度遅延
  - 同期ジョブ失敗率
- 復旧:
  - D1 バックアップ/エクスポート定期実行
  - 復元手順の演習（RTO/RPO確認）

## 9. 切替判定（Go/No-Go）案

### 9.1 Go 条件

- 主要APIの疎通・性能が stg で基準値内
- D1 と Spreadsheet の差分検証が連続で許容範囲内
- index JSON 互換テストが合格
- ロールバック手順のドライラン完了

### 9.2 No-Go 条件

- 予約整合性に影響する不具合が残存
- index JSON の鮮度/互換問題が未解消
- 復旧手順が未検証
