# /review スキル

現在のブランチの変更内容をレビューし、問題点と改善案を報告する。

## 手順

### 1. 変更差分の取得

```bash
# ブランチ全体の変更ファイル一覧
git diff main...HEAD --name-only

# 未コミットの変更がある場合はそれも含める
git diff --name-only
git diff --cached --name-only
```

変更されたファイルをすべて読み、内容を理解する。

### 2. プロジェクトルール準拠チェック

AI_INSTRUCTIONS.md の絶対ルールに違反していないか確認する。

#### 必須チェック

- `build-output/` への直接編集がないか（編集は `src/` のみ）
- `types/generated-*/` への直接編集がないか
- import 制約の違反がないか:
  - デフォルトインポート (`import foo from '...'`) → 禁止
  - 別名 (`import { foo as bar }`) → 禁止
  - ワイルドカード (`import * as foo`) → 禁止
  - 動的インポート (`import()`) → 禁止
- `Object` や `any` 型を使っていないか（具体的な Core 型を指定すべき）

#### 品質チェック

- JSDoc 型アノテーション（`@param` / `@returns`）が書かれているか
- コメントが日本語で書かれているか
- 不要な `console.log` やコメントアウトされたコードが残っていないか
- UI 追加・変更時に既存コンポーネント（`Components.*`）を使っているか
- デザイン定義（`11_WebApp_Config.js`）をハードコードせず参照しているか
- データ読み取りが `getCachedData()` 経由か
- 定数が `CONSTANTS` 経由でアクセスされているか
- 状態変更が `dispatch()` 経由か

### 3. 静的解析の実行

```bash
npm run check:smart
```

エラーがあれば内容を記録する。

### 4. レビュー報告

以下のフォーマットで報告する。

```markdown
## レビュー結果: [ブランチ名]

### 変更概要
- 変更ファイル数: N 件
- 変更内容の要約

### チェック結果

| 項目 | 結果 |
|------|------|
| ルール準拠 | ✅ / ❌ |
| 静的解析 | ✅ / ❌ |

### 問題点（あれば）

1. **[重要度]** ファイル名:行番号 — 問題の説明
   - 修正案: ...

### 改善提案（あれば）

- 提案内容と理由

### 総評

[問題なし / 軽微な修正が必要 / 要修正]
```

**修正はユーザーの確認を得てから行う。** レビュー結果の報告のみを行い、修正が必要な場合はユーザーに確認する。
