# /simplify スキル

変更されたコードをレビューし、再利用性・品質・効率の観点で問題を見つけて修正する。

## 手順

### 1. 変更差分の把握

`git diff HEAD` （未コミットの変更）または `git diff main...HEAD`（ブランチ全体の変更）で変更内容を確認する。

- 変更されたファイルの一覧を取得
- 各ファイルの差分を読み取る
- 変更の意図を理解する

### 2. 再利用性チェック

既存のコード資産が活用されているか確認する。

- **UIコンポーネント**: `Components.*`（button, modal, card 等）を使っているか。ゼロからHTMLを組み立てていないか
- **デザイン定義**: `11_WebApp_Config.js` のデザイントークンを参照しているか。色・余白・フォントをハードコードしていないか
- **定数**: `CONSTANTS` 経由でアクセスしているか。マジックナンバーやハードコード文字列がないか
- **ユーティリティ**: `08_Utilities.js` の既存関数（`convertRaw*` 等）を活用しているか
- **データアクセス**: 読み取りは `getCachedData()` 経由か。シート直接アクセスをしていないか
- **状態管理**: `dispatch()` 経由で状態変更しているか。ビューから直接状態を書き換えていないか

### 3. 品質チェック

プロジェクトルール（AI_INSTRUCTIONS.md）への準拠を確認する。

- **import 制約**: 禁止パターン（default import, `as`, `*`, 動的 `import()`）を使っていないか
- **編集禁止領域**: `build-output/` や `types/generated-*/` を直接編集していないか
- **JSDoc 型注釈**: 関数に `@param` / `@returns` が書かれているか
- **コメント言語**: コメントが日本語で書かれているか
- **命名**: 意図が明確な名前か。ファイル命名規則（数字プレフィックス）に従っているか
- **不要コード**: デバッグ用の `console.log`、コメントアウトされたコード、未使用の変数がないか

### 4. 効率チェック

コードの簡素化・最適化の余地を確認する。

- **重複コード**: 同じロジックが複数箇所にないか。ヘルパー関数に抽出できないか
- **過度な複雑さ**: ネストが深すぎないか。早期リターンで簡素化できないか
- **不要な処理**: 使われていないコードパスや冗長な条件分岐がないか
- **単一責任**: 1つの関数が複数の責務を持っていないか

### 5. 修正の実施

発見した問題を修正する。

- 各問題の修正理由を明確にする
- **振る舞いを変えない修正を優先する**（機能変更が必要な場合はユーザーに確認）
- 小さな単位で修正を行う

### 6. 検証

修正後にプロジェクトの品質チェックを実行する。

```bash
npm run check:smart
```

エラーがあれば修正し、すべてのチェックが通ることを確認する。
